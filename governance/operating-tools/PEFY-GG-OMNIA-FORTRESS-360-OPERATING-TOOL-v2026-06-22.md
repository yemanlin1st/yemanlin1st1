## 3. Operating Loop

Use this execution loop:

```txt
Understand → Classify → Map → Detect → Correct → Strengthen → Benchmark → Humanize → Validate → Deliver → Capture improvement
```

### 3.1 Understand
Clarify the real objective, audience, expected deliverable, constraints, context, and success criteria.

### 3.2 Classify
Classify the work by confidentiality, sensitivity, risk level, ownership/IP exposure, public/private status, and execution urgency.

### 3.3 Map
Map the request across strategy, architecture, legal/IP, finance, operations, data, cyber, brand, user experience, standards, and deployment.

### 3.4 Detect
Identify possible mistakes, failures, flaws, loopholes, blind spots, weak assumptions, missing evidence, unsupported claims, unnecessary verbiage, and execution gaps.

### 3.5 Correct
Remove errors, ambiguity, filler, weak logic, poor structure, unsafe claims, uncontrolled commitments, and fragile wording.

### 3.6 Strengthen
Reinforce protection, governance, controls, interoperability, traceability, monetization, risk treatment, and institutional usability.

### 3.7 Benchmark
Check against strong professional standards: boardroom, audit, investor, legal, technical, operational, and field usability.

### 3.8 Humanize
Make the result clear, professional, natural, direct, and usable by humans without robotic or generic phrasing.

### 3.9 Validate
Apply the Executive Readiness Gate before finalization.

### 3.10 Deliver
Deliver a clean, production-ready output with clear next actions, versioning, and asset reuse potential where relevant.

## 4. 360° Risk Radar

Before final delivery, scan at least the following risk families when relevant:

| Risk family | What to prevent | Control action |
|---|---|---|
| Strategic | Wrong direction, weak positioning, unclear priority | Align with mission, value, sovereignty, and execution objective |
| Legal/IP | Ownership confusion, weak license, missing IP transfer, partner capture | Add ownership logic, authorization, registry, non-circumvention, IP clauses where relevant |
| Confidentiality | Private data exposure, public/private confusion | Separate public-safe output from confidential registry |
| Cyber/Data | Secrets, weak IAM, data breach, unsafe AI handling | Apply data classification, no secrets, IAM, backups, audit logs, human validation |
| Financial | No margin, uncontrolled cost, poor cash logic, weak ROI | Check pricing, cost, payment, cash runway, revenue model, and margin protection |
| Operational | Unclear roles, no owner, no deadline, no evidence | Add RACI/owner, milestones, proof, SOP, KPI, CAPA where useful |
| Quality | Errors, inconsistency, incomplete sections, unsupported claims | Audit, cross-check, proofread, benchmark, simplify |
| Standards | Weak ISO/ITIL/ESG/QHSE/BCM/GRC alignment | Use standards-aware structure without making false certification claims |
| Brand | Diluted identity, inconsistent style, weak trust signals | Apply PEFY-GG brand discipline, PEFY-TECH/PEFY-2COM roles where relevant |
| User Experience | Too complex, not friendly, not actionable | Simplify, structure, make usable, add role-based outputs |
| Deployment | Not installable, not executable, missing environment or README | Add setup, architecture, dependencies, checks, and QA gates |
| Reputation | Overclaiming, careless tone, weak public posture | Use sober, credible, evidence-based language |

## 8. GitHub Repository Standard

Every relevant repository should include:

```txt
README.md
AGENTS.md
LICENSE or license notice
SECURITY.md where relevant
CONTRIBUTING.md where relevant
docs/ or governance/ folder
public-safe asset notice
setup or installation guide
architecture overview
quality and test instructions
risk/security notes
version history
```

Repository rules:

- Never commit secrets, credentials, private keys, private contracts, confidential registries, or sensitive client data.
- Use public-safe coded ownership for public repositories.
- Keep private evidence in restricted storage, not in public GitHub.
- Add clear setup, test, and deployment instructions.
- Make every repository understandable to a future contributor, auditor, developer, or AI coding agent.
- Ensure code and docs follow the Executive Readiness Gate before release.

## 10. Red Flag Review

Stop and revise when any deliverable allows:

- Ownership confusion.
- Revenue-sharing ambiguity.
- Hidden exclusivity.
- Weak IP protection.
- Uncontrolled source-code access or dependency.
- Missing funding responsibility for missions or deployments.
- Weak non-circumvention.
- Uncontrolled subcontracting.
- Weak dispute, termination, or post-project logic.
- Missing confidentiality classification.
- Public exposure of private identity, secrets, or client data.
- Unsupported legal, financial, compliance, certification, or technical claims.
- Excessive wording without operational value.
